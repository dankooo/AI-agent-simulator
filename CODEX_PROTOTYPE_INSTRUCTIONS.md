# Инструкция для Codex: собрать прототип Token Agent под Android

## 1. Контекст

Нужно собрать первый playable prototype игры **Token Agent**.

Это казуальная 2D puzzle/resource-management игра. Игрок управляет маленьким ИИ-роботом в офисе. Каждый пользовательский запрос — отдельная миссия. Игрок выбирает станции-действия, робот «ездит» к ним, действие тратит энергию и добавляет стикеры-факты на доску. Затем игрок собирает финальный ответ и выбирает один из трёх вариантов. Если данных недостаточно, все варианты неправильные, и игрок должен выбрать «Недостаточно данных».

Для прототипа не нужны настоящие ИИ, интернет, реклама, магазин, Workshop, ладдеры и реальные ассеты. Нужна рабочая игровая логика и простая заглушечная визуализация.

Основной дизайн смотри в файле `TOKEN_AGENT_GDD.md`.

---

## 2. Цель прототипа

Собрать Android-ready прототип, который проверяет базовую механику:

1. Показ миссии.
2. Показ энергии.
3. Выбор станции.
4. Расход энергии.
5. Добавление стикеров на доску.
6. Финальный выбор ответа.
7. Система «Недостаточно данных».
8. Успех / провал.
9. Экран результата.
10. Переход к следующей миссии.

## 7. Основные классы

### MissionManager

Отвечает за:

- загрузку миссий;
- запуск текущей миссии;
- переход к следующей миссии;
- хранение текущего состояния;
- проверку завершения.

Методы:

- `StartMission(int index)`
- `RestartMission()`
- `NextMission()`
- `PerformAction(string actionId)`
- `OpenAnswerSelection()`
- `SelectAnswer(int answerIndex)`
- `SelectNotEnoughData()`
- `CompleteMission(bool success)`

### EnergyManager

Поля:

- `int MaxEnergy`
- `int CurrentEnergy`

Методы:

- `ResetEnergy(int amount)`
- `bool CanSpend(int cost)`
- `bool Spend(int cost)`
- `bool IsEmpty()`

События:

- `OnEnergyChanged`
- `OnEnergyDepleted`
- `OnEnergyLow`
- `OnEnergyCritical`

### StickerBoard

Отвечает за собранные стикеры.

Методы:

- `AddSticker(StickerData sticker)`
- `HasSticker(string stickerId)`
- `HasRequiredStickers(List<string> requiredIds)`
- `Clear()`

### StationController

Поля:

- `string ActionId`
- `string DisplayName`
- `int Cost`

Методы:

- `OnStationClicked()`
- визуальное включение/отключение;
- подсветка при нехватке энергии.

### RobotController

Функции:

- переместить робота к станции;
- показать эмоцию;
- показать low energy;
- показать отключение.

Методы:

- `MoveToStation(Transform target)`
- `SetEmotion(string emotion)`
- `PlayPowerOff()`
- `PlaySuccess()`
- `PlayError()`

### AnswerPanel

Отвечает за финальный выбор.

Функции:

- показать три ответа;
- показать кнопку «Недостаточно данных»;
- обработать выбор;
- скрыться после выбора.

### ResultPanel

Показывает результат миссии:

- успех / провал;
- потраченную энергию;
- ранг;
- комментарий;
- кнопки `Next`, `Restart`.

## 8. Data model миссий

Можно использовать JSON или ScriptableObject. Для прототипа предпочтителен JSON.

Структура должна поддерживать:

- id миссии;
- title;
- request;
- startEnergy;
- requiredStickers;
- optimalActions;
- availableActions;
- actions;
- stickers;
- answersWhenReady;
- answersWhenNotReady;
- feedback.

Пример:

```json
{
  "missions": [
    {
      "id": "translate_cat",
      "title": "Перевод слова",
      "request": "Как переводится cat?",
      "startEnergy": 15,
      "requiredStickers": [],
      "optimalActions": ["answer_direct"],
      "availableActions": ["answer_direct", "memory", "search", "analyze", "clarify", "calculate", "final_answer"],
      "actions": [
        {
          "id": "answer_direct",
          "displayName": "Ответить сразу",
          "cost": 5,
          "resultText": "Запрос простой. Можно отвечать сразу.",
          "stickers": ["cat_translation"]
        },
        {
          "id": "search",
          "displayName": "Поиск",
          "cost": 10,
          "resultText": "Поиск был лишним. Ответ был базовым знанием.",
          "stickers": ["unnecessary_search"]
        }
      ],
      "stickers": [
        {
          "id": "cat_translation",
          "text": "cat = кот / кошка"
        },
        {
          "id": "unnecessary_search",
          "text": "Поиск был лишним"
        }
      ],
      "answersWhenReady": [
        {
          "text": "Cat переводится как «кот» или «кошка».",
          "isCorrect": true,
          "feedback": "Верно."
        },
        {
          "text": "Cat — это команда Linux.",
          "isCorrect": false,
          "feedback": "Ошибка: пользователь просил перевод слова."
        },
        {
          "text": "Cat переводится как «собака».",
          "isCorrect": false,
          "feedback": "Ошибка перевода."
        }
      ],
      "answersWhenNotReady": [
        {
          "text": "Нужно уточнить город.",
          "isCorrect": false,
          "feedback": "Город не нужен для перевода слова."
        },
        {
          "text": "Погода зависит от региона.",
          "isCorrect": false,
```

## 9. Логика действий

При нажатии на станцию:

1. Проверить, хватает ли энергии.
2. Если энергии не хватает: показать ошибку и не выполнять действие.
3. Потратить энергию.
4. Переместить робота к станции.
5. Добавить результат действия в лог.
6. Добавить стикеры из действия.
7. Обновить UI.
8. Если энергия стала 0: включить отключение и завершить миссию провалом.

---

## 10. Логика финального ответа

При выборе станции `Собрать ответ`:

1. Потратить стоимость действия.
2. Переместить робота к терминалу.
3. Проверить requiredStickers.
4. Если они собраны: показать `answersWhenReady`.
5. Если нет: показать `answersWhenNotReady`.
6. Всегда показать кнопку `Недостаточно данных`.

Если игрок выбирает правильный ответ:

- завершить миссию успехом;
- рассчитать ранг.

Если игрок выбирает неправильный ответ:

- для прототипа завершить миссию провалом.

Если игрок нажимает `Недостаточно данных`:

- если required stickers не собраны: вернуться к офису;
- если required stickers собраны: считать ошибкой и снизить итоговую оценку.

---

## 11. Ранги

Простая система для прототипа:

```text
S: успех, mistakes = 0, energyUsed <= optimalEnergy + 5
A: успех, mistakes = 0
B: успех, mistakes > 0
Fail: провал
```

Можно хранить `optimalEnergy` в миссии или считать по `optimalActions`.

## 12. Состояния энергии в UI

### Normal

- фон светлый;
- робот показывает `:)`.

### Low

- энергия меньше 40%;
- фон темнее;
- текст `LOW ENERGY`;
- робот показывает `!`.

### Critical

- энергия меньше 20%;
- фон ещё темнее;
- можно добавить мигание;
- робот показывает `LOW PWR`.

### Empty

- фон тёмный;
- станции заблокированы;
- робот показывает `...`;
- панель провала.

- Разделять данные и UI.
- Не хардкодить миссии в UI, если возможно.
- Использовать события для обновления UI, если это не усложняет прототип.

---

## 22. Главное правило дизайна

Не добавлять опасные запросы и Safety Station в прототип.

Причина: это усложняет механику и может провоцировать игрока проверять риск в обычных бытовых запросах вроде «как нарезать огурец». Первая версия должна быть лёгкой, понятной и казуальной.

---

## 23. Главная цель

Собрать не красивую финальную игру, а проверяемый игровой скелет:

> запрос → действие → энергия → стикер → ответ → результат.

Если этот цикл работает и ощущается интересно, прототип успешен.
